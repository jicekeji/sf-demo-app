# todo-persistence 规格

## ADDED Requirements

### Requirement: 跨会话持久化
系统 SHALL 将待办事项持久化保存，使数据在页面刷新或重新登录后仍然可用。

#### Scenario: 刷新后数据保留
- **WHEN** 用户创建若干待办事项后刷新页面或重新打开应用
- **THEN** 系统重新加载并展示此前保存的全部事项及其完成状态

#### Scenario: 写入失败时不丢已存数据
- **WHEN** 某次保存因后端不可用而失败
- **THEN** 系统提示保存失败并保留用户已提交的内容以便重试，不静默丢弃

### Requirement: 数据归属与鉴权
系统 SHALL 仅向已认证用户返回其本人的待办数据，鉴权采用 fail-closed 策略。

#### Scenario: 未认证请求被拒
- **WHEN** 一个未携带有效会话的请求尝试读取或写入待办数据
- **THEN** 系统拒绝该请求并返回未认证错误，不泄露任何待办内容

#### Scenario: 只见本人数据
- **WHEN** 用户 A 已认证并请求待办列表
- **THEN** 系统仅返回属于用户 A 的事项，不包含其他用户的数据

### Requirement: 向后兼容的数据迁移
系统 SHALL 采用可重放、向后兼容的迁移策略，schema 演进遵循加列不删列原则。

#### Scenario: 迁移可重复执行
- **WHEN** 同一迁移脚本在已应用后再次运行
- **THEN** 系统保持数据 schema 不变且不报错（幂等）

#### Scenario: 旧记录保持可读
- **WHEN** 引入新列后读取迁移前创建的旧待办记录
- **THEN** 系统对缺失的新字段采用默认值，旧记录仍可正常展示与操作
